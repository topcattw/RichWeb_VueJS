## 背景脈絡

這個 change 要在沒有建置流程的前提下，提供一個可直接用瀏覽器開啟的單頁大富翁遊戲。技術約束是使用 HTML、Vue 3 Global Build 與 Bootstrap 5 CDN，因此實作必須維持簡單的檔案結構，同時保留足夠的可讀性與擴充性。遊戲本身以單機輪流操作為前提，不依賴任何後端或持久化儲存。

## 目標 / 非目標

**目標：**
- 提供可玩的單機大富翁 MVP，支援 2 到 4 名玩家。
- 以資料驅動的方式定義棋盤格與事件，讓規則調整集中在設定資料。
- 讓 UI 同時顯示棋盤、玩家資訊、當前回合與事件紀錄。
- 以 Vue 管理狀態與互動，以 Bootstrap 5 快速建立可用且響應式的版面。

**非目標：**
- 不實作連線對戰、AI 玩家或存檔機制。
- 不實作拍賣、交易、蓋房子、抵押、連擲額外回合等完整官方規則。
- 不導入打包工具、路由器、狀態管理套件或測試框架。

## 設計決策

1. 使用三個靜態檔案作為應用程式結構：`index.html` 放版型掛載點與 CDN、`styles.css` 放視覺樣式、`app.js` 放 Vue 應用程式與遊戲規則。
理由：符合 Global Build 的使用方式，也避免把 UI、樣式與規則全部塞進單一 HTML 檔造成維護困難。
替代方案：全部內嵌在 HTML 中更快，但後續擴充與除錯成本較高。

2. 棋盤採用資料陣列描述，每格包含類型、名稱、價格、租金與事件處理欄位。
理由：地圖、事件與經濟邏輯可以透過同一套渲染與結算流程處理，新增格子不需要改動 UI 結構。
替代方案：用大量條件判斷逐格編碼，初期較直觀，但不利於調整規則。

3. 遊戲狀態集中在 Vue 的單一 root state，包含玩家列表、棋盤、目前回合、遊戲日誌與 winner。
理由：這個專案規模不需要額外狀態管理；單一狀態來源足以支援計算屬性與畫面更新。
替代方案：拆成多個 Vue app 或外部狀態物件，但對目前規模是額外複雜度。

4. 規則採用簡化版大富翁：經過起點加薪、地產購買與收租、機會格加減金額、稅金格扣款、休息格無事發生。
理由：能保留大富翁的核心回饋循環，同時把實作控制在一次可完成的範圍內。
替代方案：完整複製官方規則，但功能面會膨脹，且與目前無建置工具的示範目標不成比例。

5. 版面上使用 Bootstrap 元件做資訊層次，棋盤格採 CSS Grid 客製視覺，而不是依賴純 Bootstrap 網格排版。
理由：Bootstrap 適合表單、卡片與操作區，但棋盤需要固定空間感與高辨識度，自訂 Grid 更合適。
替代方案：完全使用 Bootstrap row/col，但格子排列與響應式控制會更僵硬。

## 風險 / 取捨

- [規則簡化後不等同經典大富翁] → 在介面與文件中明確定位為 MVP／簡化版，避免錯誤期待。
- [Global Build 缺少模組化機制] → 將資料、工具函式與 Vue methods 分段整理在 `app.js` 內，避免雜湊式程式碼。
- [重新整理會失去進度] → 明確接受這是無持久化範圍，先聚焦單次遊玩體驗。
- [玩家多時事件紀錄過長] → 日誌區限制高度並允許捲動，避免壓縮主要棋盤畫面。
