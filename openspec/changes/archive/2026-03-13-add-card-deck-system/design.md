## 背景脈絡

現有 Monopoly Lite 的機會格直接使用 `CHANCE_EVENTS` 陣列隨機取值，效果只有金錢增減。這種做法雖然簡單，但缺少牌堆與棄牌堆的節奏，也無法自然擴充更多事件類型。卡牌系統需要在維持單頁、單一 Vue root state 的前提下，將機會事件改成資料驅動且可視覺化揭示的流程。

## 目標 / 非目標

**目標：**
- 以牌堆與棄牌堆取代現有的機會事件隨機陣列。
- 支援至少三類卡牌效果：金錢增減、相對移動、指定位置移動。
- 在抽牌時顯示明確的卡牌標題、描述與效果結果。
- 當牌堆抽空時，自動把棄牌堆洗回牌堆。

**非目標：**
- 不加入玩家手牌、保留卡、免租卡、監獄卡或多種牌組分類。
- 不實作複雜連鎖效果編排器，也不做動畫導向的卡牌互動。
- 不處理持久化存檔或跨局保存牌堆順序。

## 設計決策

1. 將卡牌定義為資料物件陣列，每張牌包含 `id`、`title`、`description`、`effectType`、`payload` 與 `tone`。
理由：卡牌效果可用資料驅動方式擴充，避免在邏輯中散落大量特殊分支。
替代方案：直接用函式陣列處理每張卡，彈性高但不利於顯示與維護。

2. root state 新增 `chanceDeck`、`chanceDiscard` 與 `lastDrawnCard`。
理由：牌堆狀態需要與 UI 同步顯示，集中在 root state 最容易讓抽牌、棄牌與介面更新保持一致。
替代方案：把牌堆藏在閉包或模組變數中，狀態更新較難觀察。

3. 玩家踩到機會格時流程改為 `drawCard -> reveal -> resolveEffect -> discard`。
理由：這樣能明確劃分抽牌與結算階段，也方便之後加入更多效果類型。
替代方案：抽牌時立即只回傳最終結果，會讓 UI 難以顯示完整牌面。

4. 卡牌造成的移動只允許再結算一次目的地效果；若移動到另一個機會格，不再連鎖抽第二張牌。
理由：可以避免連續抽牌造成複雜遞迴與難以預測的回合流程。
替代方案：允許完整連鎖，但實作與測試複雜度會明顯上升。

## 風險 / 取捨

- [卡牌移動效果可能讓流程變複雜] → 限制移動後最多只追加一次目的地結算，避免無窮連鎖。
- [牌堆抽空後重洗若不明確，玩家會困惑] → 在事件紀錄與牌堆資訊區明確顯示重洗發生。
- [卡牌資料增加後 `app.js` 再變長] → 將牌堆工具函式與效果解析集中整理，避免與地產邏輯交錯。

## 導入計畫

1. 定義卡牌資料、洗牌工具與牌堆初始化。
2. 改寫機會格結算流程為抽牌與效果解析。
3. 新增卡牌揭示 UI 與牌堆狀態顯示。
4. 驗證抽牌、重洗、移動效果與金錢效果。
