## 背景脈絡

現有 Monopoly Lite 已有地產購買、固定租金與破產淘汰，但地產一旦買下就不再變化，導致回合中的選擇主要只剩擲骰與是否購買。房屋升級要延伸既有 `board` 與 `pendingAction` 模型，在不引入新依賴與不重寫整個回合流程的前提下，增加一層可管理的投資決策。

## 目標 / 非目標

**目標：**
- 讓每個 property tile 擁有可升級的等級、升級成本與對應租金表。
- 允許玩家在自己回合中對名下地產執行升級，並立即反映在租金與 UI 上。
- 讓操作流程維持單頁、單狀態來源與易讀的 Vue methods 結構。

**非目標：**
- 不加入旅館、抵押、拍賣、地產交換或同色組完整壟斷規則。
- 不限制每回合只能升級一次，也不處理跨回合建築隊列。
- 不處理舊存檔遷移，因為目前專案尚未導入存檔能力。

## 設計決策

1. 在 `BOARD_TEMPLATE` 的 property tile 上新增 `level`、`maxLevel`、`upgradeCostBase`、`rentByLevel`、`district` 欄位。
理由：升級規則應該跟地產資料綁在一起，這樣租金計算與畫面渲染都能以 tile 為中心處理。
替代方案：把升級資訊另外存到玩家資產表，會增加同步成本，且讓棋盤顯示更分散。

2. 升級操作採用明確的「選取一筆名下地產後升級」，而不是落地時自動升級。
理由：升級是投資決策，不應綁死在移動結果上；玩家應能在自己回合主動選擇把資金投入哪塊地。
替代方案：只允許停在自己土地時升級，規則更嚴但會讓行動窗口過窄。

3. 升級成本使用每塊地的 `upgradeCostBase` 與當前等級推導下一級成本，租金則直接從 `rentByLevel` 讀值。
理由：成本可以保留可調彈性，租金用表格查值比公式更容易平衡。
替代方案：全部用固定公式生成，實作更短，但後續調整單塊地平衡會受限。

4. 在現有 HUD 增加可升級資產清單與升級按鈕，不新增獨立 modal。
理由：目前 UI 已有側欄操作區與資產面板，直接擴充可減少互動跳轉與樣式複雜度。
替代方案：用 Bootstrap modal 顯示資產管理，但會讓回合操作分散到更多視窗層級。

## 風險 / 取捨

- [升級後租金成長過快，容易雪崩] → 先限制最大 3 級，並讓 `rentByLevel` 採保守遞增。
- [可升級資產太多時，側欄操作變擁擠] → 將升級區塊設計為簡短列表，只顯示必要資訊與主要按鈕。
- [地產資料欄位增加後 `app.js` 更長] → 將 property 相關輔助函式集中整理，避免把升級邏輯散落在多個 methods 中。

## 導入計畫

1. 擴充棋盤資料模型與 property 輔助函式。
2. 加入升級可用性判斷、成本扣款與租金查值流程。
3. 更新 HUD 與棋盤格視覺，顯示等級與升級行動。
4. 驗證升級、收租與滿級限制流程。
