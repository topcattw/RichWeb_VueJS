## 背景脈絡

目前 Monopoly Lite 的所有狀態都集中在單一 Vue root state 中，包含棋盤、玩家、回合、地產升級、牌堆與事件紀錄。這讓存檔功能很適合用「整局快照」方式處理，但也代表匯入時若直接覆蓋狀態，必須先驗證資料完整性，並清楚處理 localStorage 暫存與畫面內進度的覆蓋關係。

## 目標 / 非目標

**目標：**
- 在每次有效互動後自動將遊戲狀態暫存到 localStorage。
- 允許玩家手動將遊戲狀態匯出成 JSON 檔案，並從 JSON 檔案載回。
- 在匯入前明確提醒覆蓋目前進度與 localStorage 暫存，並要求使用者確認。
- 對匯入資料做版本與結構驗證，避免壞檔直接覆蓋狀態。

**非目標：**
- 不支援多存檔槽管理或雲端同步。
- 不直接把檔案寫入任意本機路徑，只處理瀏覽器可用的下載與上傳流程。
- 不處理跨版本 migration 工具，只先以 `schemaVersion` 驗證目前版本是否可讀。

## 設計決策

1. 使用「完整快照」方式存檔，將 root state 中必要的可變資料序列化成 JSON。
理由：目前遊戲狀態集中，完整快照最直接，也最容易保證還原後一致。
替代方案：只存部分差異值，但會提高還原邏輯複雜度與遺漏風險。

2. 自動暫存採 localStorage，手動匯出匯入採 JSON 檔下載與上傳。
理由：這是純前端專案中最相容且不需額外權限的組合。
替代方案：使用 File System Access API，可直接寫檔，但瀏覽器相容性與權限行為較不穩定。

3. 匯入流程採三段式：選檔、驗證與摘要、覆蓋確認後再套用。
理由：使用者要求匯入後除了載入，也必須提醒會覆蓋 localStorage 並徵求同意。
替代方案：選檔後立即載入，但容易誤覆蓋既有暫存與進行中的遊戲。

4. 自動暫存只在有效狀態變更後觸發，例如開局、擲骰結算、買地、升級、抽牌、回合結束、載入成功與重設。
理由：這樣既能保留最新進度，又不會在每次輸入表單文字時頻繁覆寫。
替代方案：監聽整個 state 深層變化自動存，雖然全面，但較難控制觸發時機與除錯。

5. 匯入成功後同時覆蓋記憶體中的遊戲狀態與 localStorage，並在事件紀錄加入載入成功訊息。
理由：可讓畫面狀態與自動暫存保持一致，避免重新整理後回到舊局面。
替代方案：只覆蓋畫面不覆蓋 localStorage，但會讓暫存內容與當前畫面脫節。

## 風險 / 取捨

- [匯入壞檔可能導致遊戲狀態損毀] → 載入前先驗證版本、欄位與參照關係，不通過就整體拒絕。
- [localStorage 內容被覆蓋後無法回復] → 匯入前先清楚提示會覆蓋目前進度與自動暫存，再要求確認。
- [狀態欄位增加時存檔格式容易漂移] → 引入 `schemaVersion` 並把序列化與還原集中在同一組工具函式。

## 導入計畫

1. 定義存檔資料格式、版本欄位與序列化/驗證工具。
2. 實作 localStorage 自動暫存與載入暫存流程。
3. 實作 JSON 匯出與上傳匯入流程，加入覆蓋確認。
4. 更新 UI 與說明文件，並驗證匯出、匯入與覆蓋行為。
